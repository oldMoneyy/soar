.o 文件只是编译中间产物，用于增量编译（改一个 .cu 只需重编那一个 .o 再重新链接）。 你最终需要的只有 .so 文件和 .py 文件，不需要打包任何 .o 文件。
编译目标

CMakeLists.txt 定义了 4 个编译目标 (Target)

每个 target 将一组源文件编译、链接成一个独立的 .so 共享库。
common_ops_sm90_build
→ sm90/common_ops.abi3.so
最大最核心的 target。包含绝大部分 CUDA kernel：RMSNorm、RoPE、Attention、GEMM（含 Marlin GPTQ）、MoE、Speculative Decoding、KV Cache I/O、Mamba 等。

同时编入了外部依赖：FlashInfer (norm/sampling)、Sparse Flash Attention、Fast Hadamard Transform。
~241 MB
flash_ops
→ flash_ops.abi3.so
Flash Attention 3 (Hopper 架构专用)。每种 dtype (bf16/fp16/fp8) × 每种配置 (paged/split/softcap/packgqa) 都是独立 .cu 文件，所以编译单元特别多。
~321 MB
spatial_ops
→ spatial_ops.abi3.so
Green context stream 相关功能。只有 2 个源文件，编译很快。
~100 KB
flashmla_ops
→ flashmla_ops.abi3.so
Flash MLA (Multi-head Latent Attention) kernel。来自 cmake/flashmla.cmake。
~6.5 MB

路径对照

三个目录的关系

目录 角色 内容 是否需要打包
.../sgl-kernel/csrc/ 源码目录 你写/改的 .cu 和 .cpp 文件 ❌ 不需要
.../sgl-kernel/build/ 编译输出 .o 中间文件 + 链接后的 .so ❌ 不需要 (仅增量编译用)
.../site-packages/sgl_kernel/ 安装目录 最终的 .so + .py (Python 运行时加载的位置) ✅ 从这里取 .so 打包
.../sgl-kernel/python/sgl_kernel/ Python 源码 .py 文件 (Python wrapper，调用 .so) ✅ 从这里取 .py 打包
打包提交

Wheel 打包逻辑

你的打包脚本从两个地方取文件组装成 .whl：
# 你的打包脚本做了这些事： # 1. 从 Python 源码目录取 .py 文件 /opt/.../sgl-kernel/python/sgl_kernel/*.py → whl 里的 sgl_kernel/*.py # 2. 从 site-packages 取编译好的 .so 文件 /opt/.../site-packages/sgl_kernel/sm90/common_ops.abi3.so → whl 里的 sgl_kernel/sm90/common_ops.abi3.so (241 MB) /opt/.../site-packages/sgl_kernel/flash_ops.abi3.so → whl 里的 sgl_kernel/flash_ops.abi3.so (321 MB) /opt/.../site-packages/sgl_kernel/flashmla_ops.abi3.so → whl 里的 sgl_kernel/flashmla_ops.abi3.so (6.5 MB) /opt/.../site-packages/sgl_kernel/spatial_ops.abi3.so → whl 里的 sgl_kernel/spatial_ops.abi3.so (100 KB) # 3. 从 site-packages 取 dist-info /opt/.../site-packages/sgl_kernel-0.3.20.dist-info/* # 最终产物: /opt/submission/sgl_kernel-0.3.20-cp310-abi3-linux_x86_64.whl (~250 MB 压缩后)
⚠️
注意：打包脚本用 os.walk 遍历 site-packages 下的 sgl_kernel/ 找 .so 文件。 确保 flash_ops.abi3.so 和 flashmla_ops.abi3.so 在 sgl_kernel/ 根目录下（不在子目录里）， os.walk 能遍历到它们。

不要打包 sm100/ 目录——那个用 symlink 在 prepare_env.sh 里解决。
提交包结构

submission.tar.gz 内部

submission.tar.gz 解压后: ├── prepare_env.sh ← 必须 环境构建脚本 ├── sgl_kernel-0.3.20-cp310-abi3-linux_x86_64.whl ← 你编译的 CUDA kernel │ └── (内含 4 个 .so + 所有 .py + dist-info) └── sglang/python/ ← 你修改的 sglang Python 源码 └── sglang/srt/ ├── models/minicpm.py ← 模型定义 (如果你改了) ├── layers/... └── ...
# prepare_env.sh 内容: #!/bin/bash set -e SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)" # 1. 安装你的 sglang Python 代码 (editable mode, 替换镜像内置版本) uv pip install --no-deps -e "${SCRIPT_DIR}/sglang/python" # 2. 安装你编译的 sgl-kernel wheel (替换镜像内置的 .so 文件) uv pip install --force-reinstall --no-deps "${SCRIPT_DIR}/sgl_kernel-0.3.20-cp310-abi3-linux_x86_64.whl" # 3. sm120/sm100 架构回退到 sm90 (通过 symlink) SGLK_DIR=$(python3 -c "import sgl_kernel; print(sgl_kernel.__path__[0])") ln -sf "${SGLK_DIR}/sm90" "${SGLK_DIR}/sm100"
平台执行

比赛平台拿到你的 tar.gz 后做了什么

① 启动基础镜像 平台自动

# 平台启动 docker container，内置了: # - Python 3.10 # - 原始 SGLang + sgl-kernel (预编译版本) # - CUDA Toolkit # - 模型挂载在 /models/MiniCPM-SALA

② 解压 + 执行 prepare_env.sh source 执行

# 平台解压你的提交 tar -xzf submission.tar.gz # 用 source 执行 (这样 export 的环境变量会生效) source ./prepare_env.sh # 此时 site-packages 里的 sgl_kernel 已被替换为你编译的版本 # sglang Python 代码也已替换为你修改的版本

③ 启动 SGLang 推理服务 平台自动

# 平台用 SGLANG_SERVER_ARGS 启动服务 python3 -m sglang.launch_server \ --model /models/MiniCPM-SALA \ ${SGLANG_SERVER_ARGS} # 启动时 Python 加载链路: # import sglang → 你的 sglang/python/sglang/ (editable install) # import sgl_kernel → site-packages/sgl_kernel/__init__.py # → load_utils.py 检测 GPU 架构 # → 加载 sm90/common_ops.abi3.so (或通过 sm100 symlink) # → 加载 flash_ops.abi3.so # → 加载 flashmla_ops.abi3.so # → 加载 spatial_ops.abi3.so # → torch.ops.sgl_kernel.* 注册所有 CUDA kernel

④ 正确性测试 eval_model.py

# 公开集 + 私有集，准确率必须 > 97% 否则淘汰 python3 eval_model.py --api_base http://127.0.0.1:30000 ...

⑤ 速度测试 (3 档并发) bench_serving.sh

# 测 benchmark_duration，越短越好 --max-concurrent 1 # 无并发 (权重 40%) --max-concurrent 8 # 低并发 (权重 30%) 不设 --max-concurrent # 高并发 (权重 30%)
增量编译

为什么要保留 build/ 目录

build/ 目录里的 .o 文件让你不用每次都从头编译所有 150 个文件。
# 场景：你只修改了 csrc/elementwise/fused_add_rms_norm_kernel.cu # 有 build/ 目录 (增量编译): [1/150] Building CUDA: fused_add_rms_norm_kernel.cu.o ← 只重编这 1 个 .o [2/150] Linking: sm90/common_ops.abi3.so ← 重新链接 (用旧的 84 个 .o + 新的 1 个 .o) # 耗时: ~3-5 分钟 # 没有 build/ 目录 (全量编译): [1/150] ~ [150/150] ← 全部重来 # 耗时: ~2-4 小时
💾
所以一定要备份 build/ 目录！
tar -czf /opt/sgl_kernel_build_backup.tar.gz -C .../sgl-kernel build/
这样即使环境出问题，恢复后只需增量编译改过的文件。
常见问题

"no kernel image is available" 错误解析

🔴
RuntimeError: RMSNorm failed with error code no kernel image is available for execution on the device
这个错误的含义：.so 文件里没有当前 GPU 能执行的代码。
# .so 文件内部可以包含两种东西： SASS (机器码) ← 特定架构专用，直接执行，最快 例: sm_90 的 SASS 只能在 sm_90 GPU 上跑 PTX (中间码) ← 通用，运行时 JIT 编译成当前架构的 SASS 例: compute_90a 的 PTX 可以在 sm_90, sm_100, sm_120 上 JIT # 你的 CMakeLists.txt 编译参数： -gencode=arch=compute_90,code=sm_90 # 嵌入 sm_90 SASS -gencode=arch=compute_90a,code=compute_90a # 嵌入 compute_90a PTX ← 这个是关键! # 有了 PTX，在任何 >= sm_90 的 GPU 上都能 JIT 运行 # 你的 5090 (sm_120) ✅ 比赛平台 (RTX PRO, 大概也是 Blackwell) ✅
🔍
编译完后用这个命令验证 .so 里有 PTX：
cuobjdump -lptx site-packages/sgl_kernel/sm90/common_ops.abi3.so | head
如果输出里看到 compute_90a，就说明没问题。
